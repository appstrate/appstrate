# Sidecar Protocol

Runtime detail beyond the architecture diagram in `CLAUDE.md`. Complements `docs/architecture/INTEGRATIONS_RUNTIME.md`.

### Sidecar Protocol (details beyond the architecture diagram)

- **Sidecar lifecycle**: One fresh sidecar container per run, spawned in parallel with the agent. `DockerOrchestrator.initialize()` calls `ensureImage()` for the PI + sidecar images at boot to amortise cold-pull (20–45 s) off the first run's critical path. There is no sidecar pool — empirical measurement showed the agent's own Bun cold start (~1 s) already masks warm-image sidecar boot, so pre-warming offered no user-visible latency win while carrying meaningful complexity (`/configure` endpoint, `CONFIG_SECRET`, standby network, replenish loop).
- **Parallel startup**: `pi.ts` runs sidecar setup in parallel with agent container creation + file injection via `Promise.all`. Files are batch-injected as a single tar archive before `startContainer()`. `DockerOrchestrator.createSidecar` does not await `/health` — it returns once the container is created + started, and the agent's MCP handshake retries on connection-level errors (both Node's `ECONNREFUSED` / `ENOTFOUND` / `ECONNRESET` and Bun's `ConnectionRefused` / `DNSNotFound` / `ConnectionTimeout` — see "MCP transport retry" below) with AWS full-jitter backoff (50ms→1s, 60s deadline) until the sidecar's listener comes up (#406). A non-blocking watcher races the sidecar's exit code so a sidecar that dies on boot surfaces as a structured error log (with tail of stderr) rather than a 30s MCP timeout.
- **Agent-facing surface**: every sidecar-backed capability is registered as a typed Pi tool by `runtime-pi/mcp/direct.ts`. The LLM sees the canonical first-party set — `run_history`, `recall_memory` — plus, per AFPS integration that opts into the `_meta["dev.appstrate/api"]` vendor extension, a `{ns}__api_call` outbound proxy (+ optional `{ns}__api_upload` when that auth declares `upload_protocols`) and any other tool the integration advertises, all exposed under a `{namespace}__{tool}` prefix (Phase 1.4, see "AFPS Integrations runtime" below). The agent never sees the sidecar URL.
  - `{ns}__api_call({ method, target, headers?, body?, responseMode? })` — credential-injecting proxy. `api_call` is an Appstrate VENDOR capability declared via `_meta["dev.appstrate/api"].auths.{authKey}` — ORTHOGONAL to `source.kind` (a `local`/`remote` integration with an MCP server can ALSO expose it). Single opted-in auth → tool named `api_call`; multiple → `api_call__{authToken}` per auth. Keys up to 17 characters remain verbatim; longer keys use a stable 18-character token, while the complete auth key travels separately for credential lookup. Declared synthetic names and persisted long-key aliases are reserved over same-named native MCP tools; without the extension those native names remain ordinary. Tool description, schema, and execution are all served by the sidecar's MCP `tools/call` handler, which validates against the auth's `authorized_uris` (AFPS §7.9) and injects the configured credential header server-side.
  - `run_history({ limit?, fields? })` — recent past-run metadata. Dispatched to the sidecar's MCP handler, which reads from the platform's internal run-history endpoint via the per-run token.
  - `recall_memory({ query?, limit? })` — search the agent's memory archive (rows in `package_persistence` written via the `note(content)` built-in runtime tool). Same MCP path; scoped to the caller's actor.
  - **Zero-knowledge enforcement**: after the MCP client connects, `runtime-pi/entrypoint.ts` runs `delete process.env.SIDECAR_URL`, so even the Pi bash extension cannot discover the sidecar's existence. The legacy `curl $SIDECAR_URL/…` bash pattern is fully retired — no prompt path documents it anymore.
- The sidecar exposes `/mcp` (Streamable HTTP, stateless, per-request transport) as the agent's MCP entrypoint, alongside `/health`, `GET /integrations/boot-report` (no auth, like `/mcp` — the agent's boot gate + observability source, see the AFPS Integrations runtime section), and `ALL /llm/*`. The legacy `/proxy`, `/run-history`, and `/configure` HTTP routes were retired (the latter alongside the sidecar pool — sidecars now boot fully configured via env vars). `/llm/*` is intentionally kept: the in-container Pi SDK calls `${MODEL_BASE_URL}/v1/chat/completions` and consumes the upstream stream natively, so the placeholder-substituting reverse proxy is load-bearing for primary chat completions. There is no MCP-side completion tool — sub-agent flows are handled by spawning a separate run via the platform API (each sub-task = a first-class run with its own logs, status, cost, RBAC scope, providers, and skills). For OAuth **subscription** credentials (Claude Pro/Max, ChatGPT Codex) the `/llm` oauth branch is **provider-neutral**: it applies `applyOauthBearerSwap` (`@appstrate/core/oauth-bearer-swap`) — force the real subscription bearer onto `authorization`, drop any `x-api-key`, forward every other header verbatim. It injects no provider-specific header and ensures no `anthropic-beta` flag; the Pi SDK (`@mariozechner/pi-ai`) emits each provider's request fingerprint itself. The real token is swapped in server-side, so it never enters the agent container (see `docs/architecture/SUBSCRIPTION_COMPLIANCE.md`).
- Sidecar substitutes `{{variable}}` placeholders in headers/URL/proxy (and request body if `substituteBody: true`), validates against `auths.{key}.authorized_uris` per integration.
- **SSRF layering** (initial target): three tiers, all sharing the single blocklist in `@appstrate/afps-shared/ssrf`. (1) literal floor — `isBlockedUrl` refuses IP-literal/known-internal targets everywhere; (2) DNS rebind layer — on `allow_all_uris`, no-allowlist, and glob-matched-allowlist branches, `resolveAndCheckHost` (`@appstrate/afps-shared/ssrf-dns`) resolves every A/AAAA record pre-flight and refuses if ANY lands in a blocked range (fail-closed; unresolvable → 502, blocked → 403). The connection is then delegated to `fetch` (re-resolves) — fail-closed defence-in-depth with a residual TOCTOU, same posture as the MITM upstream fetch and the platform CIMD guard; (3) resolve-and-pin — the raw-socket egress paths (forward proxy, egress listener) connect to the pinned resolved IP, fully closing the rebind window. **Literal-allowlist exemption**: an `authorized_uris` entry whose host segment is wildcard-free pins that exact host as operator-declared topology — an internal-resolving on-prem host stays reachable; a glob-host entry (`https://**`) never exempts, the concrete host being agent-chosen (`hostLiterallyAllowlisted` in `@appstrate/afps-runtime`). The CLI's `preflightUrl`/`guardedFetch` applies the identical branches.
- **Transparent egress for `delivery.env` integrations (#779)**: the plain CONNECT egress listener assumes a proxy-cooperative HTTP client, but most third-party MCP servers ship clients that ignore `HTTPS_PROXY` (undici/`fetch`) or speak non-CONNECT forward-proxy for HTTPS (axios → 405). In docker mode the sidecar therefore also mounts a per-run **transparent plane**: a UDP :53 responder (`integration-dns-responder.ts`) answers every external A query with the sidecar's own IP on the per-run bridge, and SNI-passthrough splicers on :443/:80 (`integration-transparent-listener.ts`) peek the TLS ClientHello SNI (or plain-HTTP `Host`), apply the **identical SSRF floor** (literal + resolve-and-pin), and blind-splice raw TCP — no TLS termination, no CA, cert pinning keeps working. Runner containers whose egress is the plain CONNECT kind get `--dns <sidecarIP>` (Docker's embedded DNS still serves the `sidecar` alias locally and only forwards external lookups); MITM-delivery runners are excluded (splicing would bypass credential injection). The platform grants the sidecar `net.ipv4.ip_unprivileged_port_start=0` (netns-scoped sysctl, `CapDrop: ["ALL"]` intact) only when the run declares integrations. Setup failure degrades gracefully to the proxy-env-only contract. Process mode (incl. Firecracker guests) is unchanged — no per-run network to intercept. Requires Docker ≥ 27 for embedded-DNS forwarding on `internal` networks.
- **Redirect handling** (`fetchFollowingRedirectsCapturingCookies`): manual 30x follow on the buffered path to capture `Set-Cookie` from every hop (#473). Three defence-in-depth rules apply to every hop (#475): (1) per-hop SSRF blocklist refuses redirects to loopback/RFC1918/link-local/metadata regardless of `allowAllUris`; (2) per-hop `authorized_uris` check refuses off-allowlist redirects with a structured 403; (3) hybrid credential strip — Bearer survives intra-allowlist cross-host hops (Dropbox `api ⇄ content`, Twilio `api ⇄ lookups ⇄ verify`), `allowAllUris: true` falls back to WHATWG origin-strip. Userinfo + fragment are stripped from every redirect `Location` before policy checks and before re-issuing the fetch (no attacker-injected basic-auth on the next hop). Streaming bodies skip the manual follower (initial-URL allowlist still bounds the surface).
- **Upstream metadata envelope** (`_meta["dev.appstrate/upstream"]`): every `{ns}__api_call` `CallToolResult` carries `{ status, headers, finalUrl? }`. `status: 0` = sidecar preflight failure (no upstream contact). `headers` = allowlisted projection (no Set-Cookie, no Authorization). `finalUrl` = post-redirect terminal URL, sanitised per WHATWG Fetch (userinfo + fragment stripped) — use for OAuth Authorization Code / CAS `?ticket=…` / magic-link extraction (#471). Omitted on preflight failures.
- **Proxy cascade**: Outbound requests route through proxies in priority order: agent-supplied `proxyUrl` arg → `PROXY_URL` env var (infrastructure). Agent-level and org-level proxy config is resolved by the platform before container creation.
- **Transparent pass-through**: Sidecar forwards upstream responses as-is (HTTP status code + body + Content-Type). Truncation (>50KB) signaled via `X-Truncated: true` header. Sidecar-specific errors (credential fetch, URL validation) return JSON `{ error }` with 4xx/5xx status.
- **Prompt building**: `renderPlatformPrompt()` (in `@appstrate/afps-runtime`) generates the 9-section preamble (User Input, Configuration, Pinned slots, …) + appends raw `prompt.md`. No Handlebars. Memory archive and run history are NOT rendered as prompt sections — they are exposed as MCP tools (`recall_memory`, `run_history`) whose descriptions self-document the capability.
- **Output validation**: If `output.schema` exists, it is injected into the agent container via `OUTPUT_SCHEMA` env var and becomes the input schema of the `output` runtime tool (`runtime-tool-defs.ts`) — the model sees the exact JSON Schema in the tool definition and the call is AJV-validated in-container. At ingestion, AJV re-validates the result against `output.schema` (replace-on-emit semantics, `tool-output@2.0.0`). On mismatch, the run is marked **failed** (no silent warning).
- **Unified memory surface (#273)**: a single `package_persistence` table (Drizzle) with `(actor_type, actor_id)` scope (`member` / `end_user` / `shared`) and two orthogonal attributes `(key, pinned)` covering 3 quadrants: archive (key=null, pinned=false — written via `note(content)` built-in runtime tool, read via `recall_memory` MCP tool), pinned memo (key=null, pinned=true), and pinned named slot (key=`<string>`, pinned=true — written via `pin(key, content)` built-in runtime tool, read by the prompt builder). The platform exposes `pinned: Record<string, PinnedSlot>` on `RunResult`; `runs.checkpoint` (DB) is preserved but only as the per-run snapshot consumed by the `run_history` MCP tool. Legacy `runs.state` + `package_memories` table + the standalone `/memories` REST surface + the `memories:read|delete` permission are gone — superseded by `package_persistence` and the unified `persistence:read|delete` permission. REST surface: `GET /api/agents/{scope}/{name}/persistence?kind=pinned|memory` (list/aggregator), `DELETE …/persistence?kind=…` (bulk), and `DELETE …/persistence/memories/:id` + `DELETE …/persistence/pinned/:id` (per-row).

### Reserved `_meta` vocabulary (`dev.appstrate/*`)

All first-party MCP `_meta` keys are reverse-DNS namespaced under `dev.appstrate/` (AFPS §2.2 / Appendix B). They split into two surfaces — tool-**descriptor** markers (stamped on `tools/list` entries) and tool-**result** keys (carried on `CallToolResult._meta`). Every key name below is verified against its code constant; the constant is the source of truth.

**Tool-descriptor markers** (`packages/mcp-transport/src/tool-meta.ts`) — set by the sidecar on trusted descriptors and consumed by the agent runtime (`runtime-pi/mcp/direct.ts`) to route by capability instead of pattern-matching the `{ns}__api_call` / `{ns}__api_upload` tool name. Capability detection is by key presence; the payload is load-bearing for sibling identity. `McpHost` strips both privileged markers from non-trusted MCP descriptors before they reach the runtime, so a third-party server cannot claim either capability.

- `dev.appstrate/api-call` (`API_CALL_TOOL_META_KEY`) — marks the generic credential-injecting `api_call` tool and carries `{ tool_key }`, the canonical auth-scoped key (`api_call` or `api_call__<authToken>`). The runtime wires the `fromFile` / `responseMode.toFile` resolvers onto it.
- `dev.appstrate/api-upload` (`API_UPLOAD_TOOL_META_KEY`) — marks the resumable `api_upload` tool and carries `{ api_call_tool_key }`. The runtime pairs it with the unique `api_call` carrying that key **inside the same McpHost namespace**, then wires the chunked-upload orchestrator onto it.

**Tool-result keys**:

- `dev.appstrate/upstream` (`UPSTREAM_META_KEY`, `packages/mcp-transport/src/upstream-meta.ts`) — set by the sidecar's `api_call` handler, consumed by the runtime-side parser `readUpstreamMeta()` (`runtime-pi/mcp/upstream-meta.ts`). Carries `{ status, headers, finalUrl? }`. **MUST be present on every `api_call` `CallToolResult`, success or failure** — the runtime throws `sidecar protocol violation` on a missing payload (the pre-`_meta` compat shim is dead code; sidecar and runtime ship from the same source tree). **`status: 0` means "no HTTP request happened"**: a sidecar pre-flight failure (credential fetch failure, URL not in `authorized_uris`, body too large) ships `status: 0`, `headers: {}` so the runtime can distinguish "no upstream contact" from "upstream returned 5xx" without relying on the absence of `_meta`. `headers` is re-filtered through `UPSTREAM_HEADER_ALLOWLIST` on parse (defence-in-depth against a compromised sidecar).
- `dev.appstrate/token-budget` (`TOKEN_BUDGET_META_KEY`, `runtime-pi/sidecar/mcp.ts`) — set by the sidecar on every text-path tool result. Carries `{ estimatedTokens, consumedTokens, runBudgetTokens, inlineCapTokens, decision, reason }`. **Informational: no production reader today.** The spill decisions it reports are enforced and logged sidecar-side, so a client that drops `_meta` loses telemetry, not behaviour — do not make it load-bearing without first giving it a reader. Distinct key from `dev.appstrate/upstream` so one result can carry both without collision. Full knob/reason semantics: `runtime-pi/sidecar/README.md` → "Token-aware context budgeting".
- `dev.appstrate/events` (`RUNTIME_TOOL_EVENTS_META_KEY`, `packages/core/src/runtime-tool-defs.ts`) — the trusted channel the platform's own runtime tools (`output`/`log`/`note`/`pin`/`publish_document`) use to surface canonical run events; consumed by the platform's event pipeline. A third-party integration has no legitimate reason to set it, so the sidecar's `McpHost` strips it from every third-party upstream result (`stripForgedRuntimeEvents`, `runtime-pi/sidecar/mcp-host.ts`) to prevent run-event forgery.

Not to be confused with the **manifest-side** `_meta` vendor extensions in the same namespace (`dev.appstrate/api`, `dev.appstrate/mcp-server`, `dev.appstrate/workspace`) — those live in AFPS package manifests, not on the MCP wire; see `docs/architecture/INTEGRATIONS_RUNTIME.md` and the AFPS spec.

Per the MCP spec, clients ignore unknown `_meta` keys — adding a key to this vocabulary is wire-compatible; renaming one is a protocol break on both Docker images at once.

### The sidecar `/mcp` is not a general-purpose MCP server

It is host-header-gated to `sidecar` / `127.0.0.1` / `localhost` and has exactly one supported client: the agent runtime's MCP bridge (`runtime-pi/mcp/direct.ts`). Several `api_call` capabilities are **declared** in the tool descriptor but **implemented in that bridge**, because the sidecar has no view of the agent workspace:

| Capability                  | Declared                                             | Implemented                                              | Against another client                                                         |
| --------------------------- | ---------------------------------------------------- | -------------------------------------------------------- | ------------------------------------------------------------------------------ |
| `body.fromFile`             | `CREDENTIAL_PROXY_INPUT_SCHEMA`                      | `runtime-pi/mcp/api-call-body-resolver.ts`               | Hard error — the sidecar refuses an unresolved `{fromFile}` as "a runtime bug" |
| `responseMode.toFile`       | same (description says "Resolved agent-side")        | `runtime-pi/mcp/api-call-response-resolver.ts`           | Silent no-op — the handler never reads the field                               |
| `{ns}__api_upload`          | full descriptor, so it cannot drift from the runtime | `runtime-pi/mcp/api-upload-extension.ts`                 | Hard error — the sidecar handler is advertise-only                             |
| workspace resource spilling | not declared                                         | `packages/runner-pi/src/runtime-tools/resource-spill.ts` | Resources stay as `resource_link`s, never materialised                         |

This is a deliberate trade-off — the descriptors live next to the credential proxy so they cannot drift from it — and it is safe only because the bridge is the sole client. **Do not point a generic MCP client at this server**, and do not add a capability that depends on the bridge without adding a row here.

The converse rule governs the result shape. `api_call` declares **no `outputSchema`** and attaches **no `structuredContent`** on the inline path (issue #876). Declaring an `outputSchema` makes the MCP SDK client demand a `structuredContent` on every non-error result; the sidecar then attached `{ status }`, which does not mirror `content`, and clients that prefer `structuredContent` (Claude Code, VS Code) handed the model `{"status":200}` and discarded the response body. The rule: **anything the model must see lives in `content`.** `_meta` is for the bridge, and `structuredContent` is only ever a verbatim mirror of a `content` text block.

### MCP transport retry: Bun-side error codes (#critical for the integration-runtime race)

`packages/mcp-transport/src/client.ts` retries the initial `client.connect(transport)` handshake on connection-level failures (sidecar boot race, DNS not yet propagated, etc.). The retry classifier matches `err.code` against `RETRYABLE_CODES` — and that set **must include both Node and Bun naming conventions**:

- **Node / undici**: E-prefix UPPER_SNAKE_CASE (`ECONNREFUSED`, `ENOTFOUND`, `ETIMEDOUT`, …)
- **Bun fetch**: PascalCase, no E-prefix (`ConnectionRefused`, `ConnectionTimeout`, `DNSNotFound`, …)

Missing Bun aliases means `isFatalConnectError()` would classify a Bun-side `ConnectionRefused` as fatal (the `TypeError` fallback fires when no recognised retryable code is found) and bypass the retry loop entirely. The agent container would then fail its very first MCP handshake against a still-booting sidecar instead of riding through the warm-path race window (~50-200ms). The container-per-integration path made this race much more visible because spawning a runner container before answering the agent's `tools/list` extended the sidecar's warm-up window.
